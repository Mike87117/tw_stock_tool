# DEVELOPMENT ROADMAP

本文件描述 `tw_stock_tool` 的長期發展方向與功能分層。它不是投資建議，也不是獲利承諾；本專案目前定位仍是「台股技術分析與研究工具」。任何進一步的交易自動化能力，都必須建立在嚴格驗證、風險控管與安全機制之上。

## 核心原則

- 目前階段仍是研究工具，重點是資料整理、技術分析、回測與驗證。
- 自動下單是長期目標，不是目前功能。
- 在自動下單前，必須完成並驗證回測、Walk Forward、Paper Trading、Risk Manager 與 Kill Switch。
- 本專案不提供投資建議，不保證獲利，也不應被視為任何買賣建議。
- 所有策略結果都只能代表歷史資料中的表現，不代表未來績效。

## 長期目標路線圖

### 1. 台股資料取得

目標是穩定取得台股價格資料，並降低單一資料源失效造成的影響。

目前方向：

- 支援 Yahoo Finance 台股資料。
- 支援 `.TW` 與 `.TWO` 自動 fallback。
- 支援 TWSE / TPEx 官方資料作為 fallback。
- 支援 cache，降低重複下載與 rate limit 風險。
- 提供 smoke check 工具，手動確認官方資料來源與價格資料來源是否可用。

後續重點：

- 持續強化資料錯誤訊息。
- 建立更完整的資料品質檢查。
- 清楚標記資料來源與調整價狀態。

### 2. 技術指標分析

目標是提供穩定、可測試、可重用的技術指標計算能力。

目前方向：

- MA、RSI、MACD、KD、Bollinger Band、ATR、OBV 等常見指標。
- 技術分數與 BUY / WATCH / HOLD / SELL 訊號。
- 單股分析報告與圖表輸出。

後續重點：

- 持續驗證指標計算與主流券商 / TradingView 的差異。
- 保持 no look-ahead bias。
- 讓指標與訊號邏輯更容易參數化與測試。

### 3. 多股票掃描

目標是快速從股票清單中找出值得進一步研究的標的。

目前方向：

- 支援股票清單檔案與 CLI 股票輸入。
- 支援 auto stock list，自動取得 TWSE / TPEx 股票清單。
- 支援 `--stock-limit`、`--stock-sample`，避免第一次使用就全市場掃描。
- 支援多執行緒掃描與錯誤保留。

後續重點：

- 改善大量股票掃描的穩定性。
- 強化錯誤分類，例如下市、代號錯誤、資料源暫時不可用。
- 提供更清楚的 GUI 顯示與篩選體驗。

### 4. 回測

目標是讓策略可以在歷史資料上被一致且可重現地檢驗。

目前方向：

- 使用下一交易日成交，避免同日訊號同日成交造成 look-ahead bias。
- 支援手續費、證交稅、停損、停利、最大持有天數、部位大小。
- 輸出交易紀錄、績效指標與 Equity Curve。

後續重點：

- 持續測試 no look-ahead bias。
- 更清楚區分訊號日、下單日與成交日。
- 未來可加入滑價、流動性限制與成交量約束。

### 5. Parameter Sweep

目標是系統化測試不同策略參數在歷史資料上的表現。

目前方向：

- 支援 MA Cross、RSI、Score 策略參數掃描。
- 支援 CSV / Excel 輸出。
- 支援排序、top N 與錯誤保留。

重要提醒：

- Parameter Sweep 只能找出歷史資料中表現較好的參數。
- 它非常容易產生過度擬合。
- Parameter Sweep 結果不能直接視為未來最佳策略。

### 6. Walk Forward 驗證

目標是檢查 Parameter Sweep 找到的參數是否具有跨時間區間的穩定性。

目前方向：

- 將歷史資料切成 train / test 視窗。
- 在 train 區間挑選參數。
- 用 test 區間驗證參數。
- 輸出 Summary / Detail / Errors。

後續重點：

- 強化跨視窗穩定性分析。
- 增加更多風險指標。
- 將 Walk Forward 作為進入 Paper Trading 前的必要檢查。

### 7. Daily Report

目標是每天快速產生候選股票清單，協助使用者進一步研究。

目前方向：

- 支援股票清單掃描。
- 依 Signal、Score、Volume Ratio 等條件篩選候選股。
- 輸出 Summary、Candidates、All、Errors Excel。

後續重點：

- 改善候選股摘要呈現。
- 加入更清楚的風險提示。
- 與 GUI 整合，讓新手更容易使用。

### 8. Paper Trading

目標是在不使用真實資金的情況下，模擬策略每日訊號、持倉與交易紀錄。

這是進入半自動或自動交易前的重要安全階段。

必要能力：

- 每日產生模擬下單紀錄。
- 追蹤模擬持倉、現金、損益與風險暴露。
- 記錄策略訊號與實際模擬成交的差異。
- 支援暫停、重啟與完整操作紀錄。

進入下一階段前，Paper Trading 必須長時間穩定運作，並確認結果符合回測預期。

### 9. Risk Manager

目標是在任何交易自動化之前，先建立獨立的風險控管層。

必要能力：

- 單筆交易最大風險限制。
- 單日最大虧損限制。
- 最大持倉數限制。
- 單一股票最大部位限制。
- 總曝險限制。
- 連續虧損降風險機制。
- Kill Switch，可立即停止所有自動化交易行為。

Risk Manager 必須獨立於策略本身，不能讓策略繞過風控直接下單。

### 10. Broker Interface

目標是建立抽象券商介面，讓未來下單功能與特定券商 API 解耦。

必要能力：

- 查詢帳戶餘額。
- 查詢持倉。
- 查詢委託狀態。
- 建立委託。
- 取消委託。
- 記錄 API 回應與錯誤。

重要限制：

- Broker Interface 只是介面設計，不代表啟用真實下單。
- 在 Paper Trading、Risk Manager、Kill Switch 完成前，不應接入真實下單流程。

### 11. 半自動下單

目標是在系統產生建議後，由使用者手動確認才送出委託。

必要條件：

- 策略經過回測與 Walk Forward 驗證。
- Paper Trading 已長時間穩定運作。
- Risk Manager 已完成並可阻擋高風險委託。
- Kill Switch 已完成。
- 使用者必須明確確認每一筆委託。

半自動下單的定位是「輔助決策與人工確認」，不是讓系統自行決定交易。

### 12. 自動下單

自動下單是長期目標，不是目前功能。

在進入自動下單前，至少必須完成：

- 回測驗證。
- Walk Forward 驗證。
- Paper Trading。
- Risk Manager。
- Kill Switch。
- Broker Interface。
- 完整紀錄與可追溯性。
- 異常處理與人工介入機制。

自動下單階段仍必須遵守：

- 不保證獲利。
- 不提供投資建議。
- 使用者需自行承擔交易風險。
- 系統必須優先保護資金安全，而不是追求交易頻率。

## 階段關係

```text
資料取得
↓
技術指標分析
↓
多股票掃描 / Daily Report
↓
回測
↓
Parameter Sweep
↓
Walk Forward 驗證
↓
Paper Trading
↓
Risk Manager + Kill Switch
↓
Broker Interface
↓
半自動下單
↓
自動下單
```

## 結論

`tw_stock_tool` 的短中期目標是成為穩定、可驗證、可解釋的台股研究工具。自動下單雖然可以作為長期方向，但必須在資料、策略、回測、風控與人工安全機制都成熟後才可考慮。

在任何階段，本專案都不提供投資建議，不保證獲利，也不應被視為替代使用者自行判斷風險的工具。
